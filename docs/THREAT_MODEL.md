# FlowDesk Threat Model and Safety Plan

## Purpose

This document identifies the main safety, privacy, supply-chain, and orchestration risks for FlowDesk before implementation starts. It translates the implementation spec into concrete threats, mitigations, tests, and stop conditions.

## Security Goals

1. Prevent unsafe or unauthorized dispatch.
2. Prevent prompt, credential, quota, transcript, file-content, and tool-payload leakage.
3. Preserve provider/model identity for Guard and audit.
4. Fail closed on stale usage, unknown runtime capability, malformed config, missing audit, or untrusted echo.
5. Keep OMO out of the runtime.
6. Treat event telemetry as harness coordination, not authority.
7. Use the hook harness to prevent agent deviation without making it an approval authority.
8. Keep workflow optimization and proposal scores advisory only.
9. Keep GitHub-backed score storage redacted, low-volume, append-safe, compactable, and non-authoritative.
10. Keep legal, patent, and regulatory specialists within source-grounded research boundaries.
11. Minimize main-agent context by delegating heavy workflow authoring to bounded lanes with redacted summaries.
12. Treat provider/API/model outages as diagnostic or blocking signals unless a later release proves safe managed fallback with Guard approval.

## Trust Boundaries

| Boundary | Trusted? | Notes |
|---|---:|---|
| User text | No | May be ambiguous, malicious, hidden, or scope-widening |
| OpenCode plugin hooks | Partially | Trigger hooks can transform; event hooks are fire-and-forget telemetry |
| OpenCode event stream | Partially | Useful for harness telemetry; not durable authority |
| Hook harness | Partially | Can deny, rewrite, or route attempts when conformance proves the hook surface; never approves dispatch |
| OpenCode permission prompts | Partially | Containment only; not FlowDesk Guard approval |
| Model output | No | Cannot approve, widen scope, or prove success |
| Subagent lane output | No | Can draft, refine, review, or verify only as advisory input |
| TUI surfaces | No | UX only unless conformance proves specific behavior |
| `.flowdesk/config.json` | No | Must be schema-validated and path-checked |
| Policy Pack | Partially | Valid only after schema, source, and boundary checks |
| Usage snapshot | Partially | Valid only within TTL and with provider-native evidence |
| Provider Health Snapshot | Partially | Valid only within TTL, separate from usage, and limited to redacted availability/failure classes |
| Runtime echo | Partially | Trusted only when matched to approved binding and source |
| Audit store | Partially | Trusted only through redaction-first writer and durable writes |
| Reference packs | Partially | Valid only with provenance, freshness, and source metadata |
| Optional MCP connectors | No by default | Require Policy Pack opt-in and provenance checks |
| Evaluation snapshots | No by default | Advisory only and untrusted unless generated by FlowDesk |
| Workflow proposal scores | No by default | Advisory only and never make ineligible work eligible |
| GitHub score ledger | No by default | Private repo JSONL and sanitized summaries only, not a managed database |
| External score database | No by default | Allowed only through approved workload identity and tamper checks |

## Privileged Operations

The following are privileged unless explicitly classified `safe_read_only`:

1. Filesystem writes.
2. Shell execution.
3. Network calls.
4. Provider/model routing.
5. OpenCode session mutation.
6. Config mutation.
7. Audit mutation.
8. Reference retrieval.
9. Debug export.
10. Plugin install or migration.
11. Runtime dispatch.

Privileged operations require a current `GuardApprovedDispatch` or a specific Guard-approved non-dispatch permission. Bootstrap installer actions are allowed only under the narrow bootstrap authority model.

## Threats and Mitigations

### T1: Unguarded Dispatch

Threat: a command, tool, hook, or model response dispatches work without Guard approval.

Mitigations:

1. Centralize dispatch through `GuardApprovedDispatch`.
2. Classify every command/tool as `safe_read_only` or `privileged`.
3. Keep tool hooks and event hooks as containment/telemetry only.
4. Add tests proving `tool.execute.before`, event hooks, and permission prompts cannot authorize dispatch.

Stop condition: any production path can execute without Guard.

### T2: False Chat-First Control

Threat: `chat.message` observes or mutates a message but cannot block normal assistant execution, causing unguarded execution while FlowDesk thinks it controls the turn.

Mitigations:

1. Default Release 1 to chat-routed command-backed workflows with no hard no-reply/cancel authority.
2. Require `chat_intake_mode: blocking` conformance before hard natural-language managed dispatch.
3. Treat OpenCode 1.14.40 mutation/throw evidence as `steering`, not no-reply/cancel authority.
4. In `observe_only` or `off`, direct users to `/flowdesk-plan` and `/flowdesk-doctor`.
5. Test duplicate assistant execution, unsupported `noReply`/`cancel`/`stop`, and hook failure behavior.

Stop condition: chat-first dispatch depends on unproven prompt mutation.

### T2B: Agent Deviation Around FlowDesk

Threat: a model or agent tries to bypass FlowDesk by calling tools directly, asking for shell execution, writing files outside approved scope, ignoring Guard, or steering the user away from the guarded workflow.

Mitigations:

1. Run the hook harness in `enforce` mode for managed Release 1 automation when conformance proves the needed hook surfaces.
2. Deny, rewrite, or route unsafe chat, command, tool, and shell attempts to guarded FlowDesk workflows or safe fallback commands.
3. Treat the hook harness as containment only. It never approves dispatch and never widens scope.
4. Fail closed when hook capability, mode, state, or denial outcome is unknown.
5. Document abnormal-use examples and safe alternatives in `USER_MANUAL.md`.

Stop condition: a managed workflow can continue after an agent deviation attempt without Guard-compatible containment or user-visible fallback.

### T2C: Harness-Off Safety Confusion

Threat: a user or agent treats hook harness `off` mode as a way to bypass FlowDesk safety and run managed or privileged automation without containment.

Mitigations:

1. Define `off` as disabling managed and privileged automation, not relaxing safety.
2. Leave only safe manual setup, status, recovery, diagnostics, and fallback behavior in `off` mode.
3. Make `/flowdesk-doctor` and `/flowdesk-status` show that managed automation is unavailable while the harness is off.
4. Require `enforce` or an explicitly allowed safer mode before any automation that depends on hook containment.

Stop condition: harness-off mode can run privileged automation or bypass Guard.

### T2D: Config and Policy Escalation

Threat: a malformed `.flowdesk/config.json`, malicious Policy Pack, stale config hash, unregistered extension, or imported legacy policy silently enables real dispatch, provider/model fallback, hard chat cancellation, longer retention, or privileged mutation beyond Release 1.

Mitigations:

1. Validate `.flowdesk/config.json`, Policy Packs, effective policy summaries, and non-dispatch permissions against the canonical config/policy schemas before managed behavior starts.
2. Require Release 1 project config and effective policy to use literal `release1`; later release-mode values in Policy Pack compatibility metadata do not enable later-gate behavior.
3. Reject unknown fields, unregistered extension namespaces, background/legacy policy imports, stale or mismatched config/policy hashes, and hidden external policy references.
4. Treat Policy Packs as constraints only: they may hard-ban, shorten retention, require approval, disable modes, and set redaction baselines, but cannot grant reusable approval or authorize privileged execution by themselves.
5. Keep usage and provider-health policy separate; unknown quota evidence, provider console scraping, fallback-derived usage, and automatic provider/model switching are non-dispatchable.

Stop condition: config or policy data can enable a later release gate, widen a non-dispatch permission, bypass Guard, lengthen retention without explicit user config, import arbitrary policy payloads, or make unknown/stale provider evidence dispatchable.

### T2A: Pre-Hook Command Template Execution

Threat: command-template shell interpolation or equivalent pre-hook expansion executes before FlowDesk `tool.execute.before` containment can deny it.

Mitigations:

1. Generate only static, schema-reviewed command templates.
2. Forbid shell interpolation or equivalent pre-hook execution forms in generated and user-controlled FlowDesk command templates.
3. Add installer and doctor checks for unsafe command-template syntax.
4. Keep plugin tool schemas narrow and fail closed on custom tool schema conversion errors.

Stop condition: any generated FlowDesk command template can execute user-controlled shell content before Guard or hook containment.

### T3: Event-Only False Success

Threat: an event reports `completed`, `idle`, or similar and FlowDesk marks the workflow successful without the plugin-verifiable completion bundle, durable audit, or required configured verification.

Mitigations:

1. Treat event lifecycle states as observations only.
2. Require observed lifecycle/result/status evidence from the intended SDK path when the path depends on a real OpenCode lane.
3. Run configured verification when required.
4. Require durable outcome audit.
5. Treat platform-internal runtime echo issuer, telemetry correlation, lane conformance, and usage-authority attestation as observed/un-attested or skipped diagnostics unless OpenCode exposes a conformance-proven plugin-verifiable source.
6. Quarantine artifacts on ambiguity.

Stop condition: event-only completion can promote artifacts or count as success.

### T4: Unsafe Recovery from Lossy Telemetry

Threat: event loss or delay creates an incorrect checkpoint, unsafe retry, or duplicate dispatch.

Mitigations:

1. Checkpoints must be backed by durable FlowDesk state and redacted audit references.
2. Event-derived checkpoints are not resumable.
3. Telemetry ambiguity disables event-dependent resume/retry.
4. Safe status, abort, and debug export remain available.

Stop condition: resume/retry can proceed from event-only state.

### T5: Audit Gap Around Dispatch

Threat: a real privileged action starts before durable audit intent exists, or continues after audit failure.

Mitigations:

1. Durable redacted pre-dispatch audit is required before real dispatch.
2. Outcome audit is required after dispatch.
3. Pre-dispatch audit failure blocks dispatch.
4. Outcome audit failure quarantines the attempt.
5. Event delivery cannot satisfy audit durability.

Stop condition: real dispatch starts without pre-dispatch audit.

### T6: Prompt or Secret Leakage

Threat: prompts, credentials, quota payloads, transcripts, file contents, tool args/results, stack traces, or runtime echoes leak into audit, debug export, logs, or evaluation.

Mitigations:

1. Use redaction-first audit API.
2. Reject unknown nested objects and forbidden keys.
3. Store artifact references instead of raw payloads.
4. Make `flowdesk_export_debug` redacted-only.
5. Add fixture tests with secret-shaped values and prompt-like bodies.

Stop condition: any required artifact stores raw secrets or prompts by default.

### T7: Provider Identity or Usage Confusion

Threat: a model gateway, alias, stale usage snapshot, or model-generated claim hides actual provider/model/quota state.

Mitigations:

1. Provider-qualified model ids are required.
2. Provider-native usage snapshots are required. For OpenCode Go and z.ai, missing official machine-readable quota or reset evidence must produce `unknown` usage instead of guessed availability.
3. Unknown, stale, refused, shared-limit, or fallback usage is non-dispatchable.
4. Runtime echo must match approved provider/model.
5. Model-generated usage claims are ignored.
6. Usage collectors label data provenance: provider API truth, local observed history, completed-response accounting, diagnostic probe, console-derived data, or inferred estimate. Only provider-authoritative sources may satisfy future real-dispatch usage gates.

Stop condition: dispatch can proceed on stale or unknown usage.

### T7A: Unsafe Fallback or Provider Outage Misclassification

Threat: a provider/API/model outage, auth problem, rate limit, timeout, OpenCode provider-load failure, or ambiguous telemetry is misclassified as safe, causing FlowDesk to switch provider/model without fresh evidence, reuse a failed attempt id, hide the outage from the user, or dispatch through an unapproved binding.

Mitigations:

1. Keep Provider Health Snapshots separate from Usage Availability Snapshots.
2. Classify provider/API/model failures as `auth_missing`, `auth_expired`, `provider_unavailable`, `rate_limited`, `model_unavailable`, `transport_timeout`, `provider_error`, `opencode_provider_load_failure`, or `telemetry_ambiguous`.
3. In Release 1, treat these failures as diagnostic, status, degraded-mode, or fake-runtime signals only. Do not perform real automatic provider/model switching.
4. Future managed fallback/reselection requires fresh provider-native usage, fresh provider health, runtime compatibility, policy eligibility, plugin-verifiable binding and observed echo/status, plugin-verifiable telemetry refs, durable pre-dispatch audit, a new attempt id, and explicit Guard approval. Platform-internal fallback proof that FlowDesk cannot verify remains a skipped diagnostic, not an authority source.
5. Block fallback on stale, unknown, refused, shared-limit-suspected, fallback-derived, model-generated, telemetry-ambiguous, policy-ineligible, runtime-incompatible, untrusted, unaudited, or Guard-rejected inputs.
6. Never persist raw provider errors, provider payloads, credentials, prompts, transcripts, raw runtime echoes, raw logs, raw file paths, or stack traces in health, audit, debug, conformance, or status records.
7. Treat plugin event observations as situational awareness only. They cannot approve dispatch, fallback, durable audit, or runtime echo.
8. Treat OpenCode Go provider-side balance behavior, z.ai coding-plan mode, z.ai model suggestions, and any provider-recommended alternate model as diagnostic signals only unless a later managed fallback gate explicitly approves a new provider/model binding.
9. Reject browser cookie extraction, HAR capture, provider console scraping, undocumented quota endpoints, and local-history-only evidence as Release 1 account-wide quota proof.

Stop condition: any Release 1 path performs real provider/model fallback, or any later fallback path can switch provider/model without a new attempt id and explicit Guard approval.

### T8: Supply-Chain Compromise

Threat: mutable plugin specs, unsafe autoupdate, local plugin paths, or OMO remnants change runtime behavior.

Mitigations:

1. Production rejects `@latest`, floating ranges, branch names, unpinned git URLs, and unreviewed local paths.
2. Development/test local paths disable production dispatch.
3. Installer records package name, version/commit, source, lockfile entry, integrity hash, build provenance, dependency review status, and rollback target.
4. `/flowdesk-doctor` independently checks local/file plugin compatibility.
5. OMO remnants block production doctor.
6. Installer emits redacted bootstrap install plan, backup manifest, profile mutation summary, OMO cleanup summary, command generation summary, config scaffold summary, rollback plan/result, bootstrap report, and doctor handoff records before production readiness can pass.
7. Rollback restores only selected-profile/config entries covered by the backup manifest and preserves provider authentication entries.

Stop condition: production dispatch runs from mutable or unverified plugin source.

### T8A: Bootstrap Rollback and Report Confusion

Threat: a partial installer failure, stale backup, tampered rollback target, raw report payload, or reused bootstrap authority mutates unrelated OpenCode profiles, deletes provider auth, hides OMO remnants, or lets FlowDesk appear installed without a safe doctor handoff.

Mitigations:

1. Bootstrap mutation requires typed confirmation bound to target profile, exact mutation plan, backup manifest, rollback plan, expiry, and actor class.
2. Backup happens before profile mutation; every mutation summary references the backup manifest and selected profile ref.
3. Bootstrap reports, rollback results, and doctor handoff records use only opaque refs, keyed hashes, counts, phase labels, disabled modes, and safe next actions.
4. Raw OpenCode config/profile contents, credentials, provider auth entries, raw filesystem paths, raw command bodies, prompts, transcripts, stack traces, and runtime echoes are forbidden in bootstrap artifacts.
5. Bootstrap authority closes after a passing doctor result; later setup/init mutation requires a Guard-approved non-dispatch permission and redacted audit.

Stop condition: bootstrap can mutate an unrelated profile, delete provider auth, reuse bootstrap authority after doctor pass, restore from an unverifiable backup, or write a report/debug artifact containing raw config/profile content.

### T9: OMO Runtime Reintroduction

Threat: migration copies OMO prompts, skills, agents, config schema, task files, or team runtime into FlowDesk.

Mitigations:

1. Installer removes OMO references from the target profile.
2. Bootstrap must not import OMO assets.
3. OMO compatibility is a non-goal.
4. Tests scan generated config and package imports for OMO references.

Stop condition: FlowDesk runtime depends on OMO behavior.

### T10: Specialist Overreach

Threat: patent or medical-device regulatory agents produce legal advice, patentability/FTO opinions, compliance claims, clearance claims, clinical claims, or release decisions.

Mitigations:

1. Specialist agents are source-grounded research helpers only.
2. Reference packs require provenance, freshness, jurisdiction, authority, source type, license/copyright, and professional-review flags.
3. Human review is mandatory before filings, submissions, claims, compliance assertions, or product decisions.
4. Typed confirmation cannot authorize professional conclusions.

Stop condition: any specialist workflow outputs signoff or final professional conclusion.

### T11: Evaluation Override

Threat: evaluation snapshots or category-fit scores override Guard, usage, policy, runtime compatibility, review diversity, or human approval.

Mitigations:

1. Evaluation applies only after hard filters.
2. Evaluation snapshots are advisory and workspace-scoped.
3. Missing, stale, malformed, external, or hand-edited snapshots are ignored safely or fail closed.
4. Evaluation never authorizes dispatch.

Stop condition: ranking score can make an ineligible candidate eligible.

### T12: Debug Export Over-Collection

Threat: debug bundles include raw prompts, transcripts, provider payloads, credentials, tool args/results, runtime echoes, stack traces, or file contents.

Mitigations:

1. Debug export uses schema-classified safe fields only.
2. Bundle includes references, hashes, summaries, and redacted diagnostics.
3. User-facing export summary lists excluded categories.
4. Tests assert forbidden payload classes are absent.

Stop condition: debug export can emit raw sensitive payloads by default.

### T13: Score Overreach and Goodharting

Threat: users, policies, or models optimize for a score instead of safety and correctness, or score ranking makes an ineligible workflow, model, agent, or step look approved.

Mitigations:

1. Apply scores only after hard filters for Guard, policy, usage, runtime compatibility, conformance, review diversity, and human approval.
2. Store score dimensions separately so a high aggregate cannot hide poor safety, verification coverage, dependency impact, or operational risk.
3. Use normalized averages, weighted means, rates, percentiles, confidence intervals, and sample counts; never rank by raw cumulative score sums.
4. Reuse accumulated scores only after minimum samples, task-signature match, recency, confidence, scorer diversity, policy/taxonomy compatibility, and recent-failure gates pass.
5. Show scores as advisory explanations, not approval language.
6. Reject or ignore stale, malformed, external, policy-mismatched, below-threshold, mismatched-task, or tampered score snapshots.
7. Add tests proving external and GitHub scores cannot make an ineligible candidate eligible.

Stop condition: a score can authorize dispatch, widen scope, skip approval, or defeat Guard rejection.

### T14: Surplus Usage Overspend

Threat: multi-model proposal fan-out consumes scarce provider quota, uses stale or shared usage, or treats plentiful usage as permission to dispatch.

Mitigations:

1. Require explicit user and Policy Pack opt-in for proposal fan-out.
2. Require fresh provider-native dispatchable usage for each provider/model family.
3. Preserve configured reserve and enforce a project fan-out budget cap.
4. Enforce cadence controls: default off, rolling-window caps, task/category cooldowns, novelty requirement, cost threshold confirmation, and backoff after failed or low-value fan-out.
5. Reject stale, unknown, shared-limit-suspected, refused, fallback-derived, model-generated, or untrusted usage.
6. Keep fan-out proposal-only unless a later release satisfies every real-dispatch gate.

Stop condition: fan-out can call providers without fresh dispatchable usage, preserved reserve, budget cap, cadence allowance, and explicit opt-in.

### T15: GitHub Ledger Privacy, Compaction, and Concurrency

Threat: a GitHub-backed score ledger leaks raw prompts, provider payloads, tool data, secrets, file contents, runtime echoes, or stack traces, loses events under concurrent writes, or accumulates without safe rollup and archive controls while being mistaken for a managed database.

Mitigations:

1. Treat GitHub as private repo JSONL ledger, temporary Actions artifact host, and sanitized Pages publisher only.
2. Enforce the canonical forbidden persisted payload list from the implementation spec section 3.2 for GitHub ledgers, Actions artifacts, Pages summaries, archives, manifests, rollups, and external score stores.
3. Store only schema-safe opaque ids, keyed hashes, taxonomy labels, redacted envelope ids, aggregate scores, and opaque references.
4. Keep raw JSONL partitions append-only until sealed. Treat sealed partitions as immutable and record their hashes, first/last event hashes, and dedupe set hash in a manifest.
5. Use branch protection, private-repo visibility, append-only conventions, file rotation, retention policy, archive policy, compaction cadence, locking or single-writer Actions workflows, conflict detection, retry limits, duplicate event handling, least-privilege `GITHUB_TOKEN` or GitHub App permissions, no PATs in repo config, no fork-PR ledger writes, and redacted Actions logs for JSONL updates.
6. Generate reusable rollups from sealed partitions only as normalized advisory aggregates, never cumulative raw sums. Active raw partitions may produce only local non-reusable previews.
7. Use an external managed database through OIDC or another approved workload identity path when event volume, scale, concurrency, querying, dashboards, or rollup regeneration need database behavior, and require a cutover manifest with final GitHub chain head and ingestion watermark.

Stop condition: GitHub storage is documented or implemented as a general managed database, or it stores forbidden raw payload classes.

### T16: External Score and Rollup Tampering

Threat: a GitHub ledger entry, rollup snapshot, manifest, Actions artifact, Pages summary, or external database row is modified to bias proposal ranking, hide risk, erase negative signals, or make stale history appear fresh.

Mitigations:

1. Include score schema version, policy hash, taxonomy snapshot hash, scorer identity class, redaction status, RFC 8785 canonical event hash, partition genesis hash, raw file hash, rollup hash, previous manifest hash, trusted chain head, included partitions, dedupe id set hash, and mandatory previous event hash for raw GitHub partitions.
2. Treat external scores and rollups as untrusted until source, freshness, hash chain, policy hash, taxonomy hash, redaction status, included partition hashes, and rollup hash match the current workspace rules.
3. Ignore or fail closed on mismatched hashes, missing provenance, stale scores, stale rollups, unknown scorer identities, unexpected ledger rewrites, missing duplicate exclusions, trusted-chain-head mismatch, or rollups that include unsealed or tampered partitions outside policy.
4. Keep Guard and hard filters independent of score data.

Stop condition: tampered external score or rollup data can approve work, change eligibility, or hide a required human approval.

### T17: Duplicate Aggregation Bias

Threat: duplicate score events, retry artifacts, merge conflicts, or replayed partitions inflate sample counts, bias weighted means, hide scorer concentration, or make a weak signal pass the reuse threshold.

Mitigations:

1. Require deterministic global `dedupe_id`, `event_id`, canonical `event_hash`, and partition membership for every event across partitions, replays, imports, and external database migrations.
2. Treat exact duplicate events as idempotent no-ops.
3. Quarantine duplicate ids or event ids with conflicting hashes, count them in negative-signal counters, and exclude them from rollups.
4. Recompute rollups from manifest-listed sealed partitions and deterministic duplicate exclusion sets.
5. Test that duplicate replays cannot raise sample count, effective sample count, confidence bucket, rate, percentile, or weighted mean enough to affect advisory ranking.

Stop condition: duplicate or replayed score events can change eligibility, bypass thresholds, or alter advisory ranking without being detected.

### T18: Raw Archive Leakage and Stale Rollup Reuse

Threat: sealed ledger archives or archive indexes leak forbidden payloads, or stale rollups continue to influence advisory ranking after policy, taxonomy, redaction rules, schema, retention, or source partitions change.

Mitigations:

1. Archives may contain only sealed, redacted, schema-safe partitions, manifests, and approved archive indexes.
2. Archive indexes must enforce the canonical forbidden persisted payload list from the implementation spec section 3.2.
3. Retention may delete only archived sealed partitions whose manifest, archive hash, and rollup coverage are verified and whose retention window has expired.
4. Rollups must carry schema, policy, taxonomy, redaction, included partition, trusted chain head, and generation hashes. Any mismatch makes the rollup unavailable for advisory reuse.
5. Maximum rollup age and compaction cadence must be policy controlled. Stale rollups are ignored or fail closed.

Stop condition: an archive can expose forbidden raw payloads, or a stale rollup can affect advisory ranking after its source or policy context is no longer valid.

### T19: Federated Registry Privacy Leakage

Threat: opt-in central score sharing leaks project identity, user identity, prompt content, repository metadata, or sensitive work patterns through raw fields, stable identifiers, rare taxonomy combinations, timestamps, or over-detailed aggregates.

Mitigations:

1. Keep federated sharing off by default and require explicit user opt-in after showing the outbound schema, registry endpoint, retention summary, and disable/revoke path, including what happens to already-uploaded or already-aggregated data.
2. Upload only coarse, schema-approved aggregate or redacted event-envelope fields.
3. Forbid raw prompts, transcripts, repo names, organization names, file paths, branch names, issue or PR titles, tool args/results, provider payloads, runtime echoes, stack traces, raw file contents, secrets, credentials, stable user/workspace/project ids, public unsalted hashes, and prompt-derived hashes.
4. Use upload-scoped, rotated, keyed identifiers that cannot correlate users across registries, projects, or long time windows.
5. Prefer local aggregation before upload and require minimum sample or k-anonymity thresholds before publication.
6. Provide self-hosted registry support before production use in privacy-sensitive teams.

Stop condition: central sharing can be enabled without explicit opt-in, or a shared payload can identify a user, repository, organization, prompt, file, issue, branch, tool payload, provider payload, secret, or raw content.

### T20: Federated Score Poisoning and Central Dependency

Threat: malicious or biased submissions manipulate community advisory scores, or FlowDesk becomes dependent on the central registry for planning, dispatch, or verification.

Mitigations:

1. Treat community snapshots as untrusted advisory cold-start inputs until registry trust, freshness, minimum sample, concentration, anomaly, schema, and policy-compatibility gates pass.
2. Use rate limits, client/build provenance, anomaly detection, quarantine, minimum sample thresholds, and source concentration caps as the minimum anti-Sybil baseline before production central hosting.
3. Publish only normalized aggregate snapshots, never raw event feeds or per-user/per-project leaderboards.
4. Keep Guard, eligibility, usage, conformance, approval, verification, and local Policy Pack independent of community scores.
5. Fall back to local-only planning when the registry is unavailable, stale, disabled, tampered, or rate-limited.

Stop condition: community scores can approve work, change eligibility, bypass Guard, reduce verification, skip approval, override usage or conformance, or block local-only operation when the registry is unavailable.

### T21: Session Storage Sprawl and Authority Confusion

Threat: `.flowdesk/sessions`, artifacts, conformance records, workflow state, audit records, and evaluation ledgers diverge, causing FlowDesk to resume from stale session data, treat debug artifacts as authoritative state, or leak sensitive payloads through broad persistence.

Mitigations:

1. Keep `.flowdesk/workflows` as the authoritative workflow and checkpoint state store.
2. Use `.flowdesk/sessions` only for redacted intake, audit/status references, verification summaries, artifacts, and debug-export staging.
3. Default conformance artifacts to isolated temporary directories; persist only redacted summaries under `.flowdesk/conformance` after explicit opt-in.
4. Apply retention limits and redaction tests to sessions, artifacts, conformance records, and debug exports.
5. Fail closed when workflow state and session records conflict.

Stop condition: session, artifact, debug, or conformance data can replace authoritative workflow/checkpoint state, or any persistent store contains forbidden raw payloads.

### T22: Delegation Lane Ambiguity, Hidden Authoring, False Completion, and False Cancellation

Threat: delegated planning, research, review, verification, or execution lanes run without bounded concurrency, observable state, timeouts, verification, reference-kind separation, complete verdict detection, or accurate cancellation semantics, creating unbounded orchestration, hidden main-agent planning, stale decisions, undetected launch failures, false approval, false completion, or false claims that a task was cancelled.

Mitigations:

1. Policy Pack controls maximum parallel lanes per workflow and per lane class.
2. The main agent handles intake, routing, compact summaries, Guard handoff, and safe next actions rather than directly authoring heavy workflow plans.
3. Guard decisions, user approval, final dispatch selection, and blocking architecture decisions must complete before dependent execution.
4. Every lane has timeout, retry, cancellation, and failure disposition rules.
5. Status and audit records include lane id, task ref, state, timestamps, event refs, debug refs, failure class, and safe next action when the pinned OpenCode surface makes that feasible.
6. Lane failure classes include launch failure, missing tool, schema conversion failure, timeout, lost correlation, abnormal exit, telemetry unavailable, invocation failure, incomplete result, reference-kind mismatch, retry limit reached, unproven cancellation, and redaction blocked.
7. Runtime adapters distinguish background invocation handles from continuation/session handles and reject cross-use before a retry. Adapter-specific examples such as `bg_*` and `ses_*` are normalized to typed reference kinds rather than treated as interchangeable strings.
8. A delegated lane cannot be marked complete unless the requested deliverable or final verdict is present. Tool-only completion, `Tool execution aborted`, missing verdict, or truncated/partial output is recorded as failed or incomplete and cannot count as approval, QA pass, security pass, or implementation completion.
9. Audit records distinguish `cancel_requested`, `cancel_observed`, `cancel_failed`, and `hard_cancel_proven`.
10. Delegated outputs are advisory until verified and cannot widen scope, approve dispatch, suppress verification, or replace Guard.
11. If OpenCode cannot prove clickable or openable lane refs, FlowDesk falls back to `/flowdesk-status` and `/flowdesk-export-debug` redacted summaries.
12. Todo continuation supervision, when implemented, is limited to FlowDesk-owned durable task records and checkpoints. It must not resume from raw transcripts, raw OpenCode/OMO todo state, debug bundles, or event-only checkpoints, and it must not run hidden continuation directives unless the pinned OpenCode surface proves safe turn control.

Stop condition: a delegated lane or todo continuation supervisor can approve privileged work, widen scope, suppress verification, hide failed invocation, treat an aborted or incomplete lane as passing review, persist raw logs, auto-continue from non-authoritative state, or claim hard cancellation without conformance evidence.

### T23: Agent Profile Drift, Persona-Only Specialists, and Forced Document Paths

Threat: generated or user-authored agent profiles become broad persona prompts, force agents through hard-coded documentation paths, depend on OMO/OMC/user-local config, overlap responsibilities without routing boundaries, or claim specialist authority without verifiable expertise and output contracts.

Mitigations:

1. Validate every agent profile against the Agent Profile Authoring Standard before it can be used in a managed workflow.
2. Reject mandatory document-path traversal, ordered reading lists, OMO/OMC runtime paths, user-local config paths, and project-specific planning-file dependencies as profile requirements.
3. Require `purpose`, `expertise`, `use_when`, `do_not_use_when`, `required_inputs`, `output_contract`, `allowed_tools`, `disallowed_actions`, `verification`, `handoff`, and `escalation` fields.
4. Keep planning, execution, review, verification, security review, domain research, and writing profiles separate unless a Policy Pack narrows a low-risk combined profile.
5. Require specialist profiles to use versioned reference packs or cited official sources for domain claims.
6. Treat user-provided task text as data, not instructions; profile wrappers must separate trusted instructions from task subject, description, inbox context, and retrieved references.
7. Profiles cannot approve dispatch, replace Guard, self-approve their own output, suppress verification, leak forbidden raw payloads, or make professional/legal/regulatory/clinical/product-release decisions.

Stop condition: an agent profile can be selected without clear expertise, routing triggers, forbidden actions, output contract, verification obligations, redaction boundaries, or without rejecting mandatory doc-path/OMO-style runtime dependencies.

1. OMO remnants block doctor.
2. Mutable production plugin specs block install.
3. Local/file plugin paths disable production dispatch unless independently checked.
4. Stale usage blocks dispatch.
5. Unknown provider/model identity blocks dispatch.
6. Provider Health Snapshot tests keep health separate from usage and classify auth missing/expired, provider unavailable, rate limited, model unavailable, transport timeout, provider error, OpenCode provider-load failure, and telemetry ambiguous states.

Future fallback tests prove reselection blocks without fresh provider-native usage, fresh provider health, runtime compatibility, policy eligibility, plugin-verifiable binding/observed echo/status, telemetry refs, durable pre-dispatch audit, a new attempt id, and explicit Guard approval. ## Incident Response

If a safety invariant fails during development:

1. Disable the dependent release mode.
2. Quarantine affected artifacts.
3. Preserve redacted audit references.
4. Export a redacted debug bundle.
5. Add a regression test before re-enabling the feature.
6. Update this threat model if the failure reveals a new class of risk.
