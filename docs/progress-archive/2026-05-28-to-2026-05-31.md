Two clarifying analyses were completed on 2026-05-31.

(1) Gate-vs-spec fit for "trusted runtime echo". The managed-dispatch Guard boundary requires trusted runtime echo + conformance metadata. Reading the normative docs shows the spec does NOT define trusted runtime echo as cryptographic attestation: `OPENCODE_CONFORMANCE_PLAN.md` Release 2 echo/telemetry smoke states the SDK `session.prompt` response and `session.messages` expose provider/model, assistant message id, session correlation, part lifecycle, and message count, and "redacted evidence records validate against runtime echo and telemetry schemas" — i.e. trusted echo is an OBSERVED SDK-session correlation validated at schema level, with production persistence/conformance still gated. So the earlier "observed-binding" issuer design is directionally consistent with the spec's intent. The spec's safety does not come from per-echo cryptography but from (a) cross-checking many independent evidence pieces (binding + usage + usage-authority + provider health + telemetry + pre-dispatch audit + human approval), (b) redaction-first, and (c) keeping production persistence/conformance separately gated. The IMPLEMENTATION_SPEC also repeatedly states `opencode run`, shell parallel launch, model text, TUI surfaces, and event lifecycle are NOT runtime echo, and untrusted echo quarantines the result. The reviewers' "trusted-laundering" concern is therefore a strengthening signal (require observed conformance modes instead of caller assertion, require observed model identity, single-use attempt binding, shorter TTL), not a contradiction of the spec — but emitting `runtime_echo_mode=trusted` from observation still must be backed by a real observed lane and carry explicit uncertainty labels.

(2) Why the so-called "MCP" Gemini usage call fails. Correcting the framing first: `flowdesk_provider_usage_live` is NOT an MCP call. The only MCP server registered in any OpenCode config is `codegraph`; FlowDesk is loaded as a plugin, and the OpenCode logs show `service=tool.registry status=started/completed flowdesk_provider_usage_live` (a plugin tool inside the OpenCode process), never an MCP round-trip. The real difference is the runtime: the OpenCode binary is a single Bun-compiled standalone Mach-O, so the plugin tool executes under Bun, whereas the direct reproduction (`node --input-type=module` importing the same installed 0.1.17 dist) executes under Node.js. The installed 0.1.17 collector code is identical and succeeds under Node (Gemini dispatchable ~65%), and a durable Gemini usage snapshot plus a fresh sidebar cache were successfully written by a Node-side run. The provider-usage tool passes only `execFile: execFileSync` to the collector and leaves `filesystem`/`fetch` defaulted, and the tool completes with `duration=0` in the OpenCode log (no collector error surfaced), which indicates the Gemini collection path short-circuits/fails immediately under Bun rather than performing the expired-token refresh (`fetch` to oauth2.googleapis.com) and cache-package file read (`node:fs`) that succeed under Node. Conclusion: the Gemini staleness is a Bun-standalone runtime behavior difference in the usage collector (expired-token refresh / fs / execFile under the compiled Bun host), not a stale process, not a FlowDesk auth/logic defect, and not an MCP transport issue. It does not block production work that uses Claude/OpenAI, and native `gemini_quota` remains available as a cross-check. A proper fix would inject Bun-compatible `filesystem`/`fetch`/`execFile` (or otherwise harden the collector's refresh path for the Bun host) and verify inside the OpenCode runtime, not just under Node.

Event-hook based subtask orchestration design was updated on 2026-05-28 after confirming OpenCode 1.15.6 exposes both plugin `Hooks.event(input: { event })` and SDK `client.event.subscribe` SSE `/event`. The event model should be the primary lane-progress path, with status polling/watchdog scans retained only for recovery, replay, and missed-event fallback. The design maps FlowDesk child-session events to lanes through durable `agent_task_child_session` binding, treats `message.part.updated` step-finish and `session.idle` as terminal-check candidates, `session.error` as a terminal-failure candidate, `session.status: retry` as non-terminal retrying, tool-part `state:error` as a non-terminal tool issue unless followed by session-level terminal/error evidence, `permission.updated/replied` as permission-wait lifecycle input, and `session.updated` only as metadata refresh/heartbeat/terminal-check trigger rather than direct completion. A workflow wait-set/aggregate card should track expected/running/completed/failed/awaiting-permission counts with event-id/digest idempotency, attempt isolation, replay protection, and stable card ids. When every expected lane in the current attempt completes normally with usable final task results and no unresolved permission wait, no task_failed, no finalizing inconsistency, and no contract-missing result, FlowDesk should automatically advance to the next planned orchestration step without requiring the user to ask for status: for review/analysis fan-out this means starting the synthesis/summary step immediately; for execution-like or write-capable next steps it may only prepare the next-step card and require the existing Guard/confirmation boundary. If any lane fails, is partial, is awaiting permission, or has contract/inconsistency evidence, automatic continuation must stop and surface retry/abort/resume/summarize-partial choices instead. ## Release 2/2.5 Implementation Confidence and Uncertainty Policy

Release 2 and Release 2.5 work should proceed incrementally even when some OpenCode runtime conformance details are not yet fully proven. Lack of complete proof is not, by itself, a blocker when the feature can run safely behind explicit opt-in gates, fail-closed checks, durable evidence capture, and clear doctor/status reporting. Uncertain behavior must be documented as an evidence limitation and kept out of default Release 1 authority claims.

### Expected Implementable Areas

1. Durable production-path evidence persistence for usage authority, runtime echo, telemetry correlation, configured verification results, approval decisions, and pre-dispatch audit records is implementable using the existing `session-evidence.ts`, state-store, audit, and validator patterns.
2. A fail-closed production approval state machine is implementable as a core contract before dispatch is enabled; it should default to deny and only later permit conditional dispatch when required evidence is present.
3. Doctor-visible production enablement states are implementable: `disabled`, `blocked`, `configured`, `approved`, and `dispatch-capable` should be represented as diagnostic state, not as implicit dispatch authorization.
4. External auth/provider policy and doctor gating are implementable with conservative provider/model/account/auth-scope binding checks and explicit non-dispatchable diagnostics when evidence is missing or stale.
5. Configured verification integration is implementable by requiring production-path verification config to be read, validated, persisted, and treated as blocking only for the guarded dispatch path.
6. Same-model multi-perspective reviewer separation is implementable at the FlowDesk artifact layer by storing distinct perspective bindings, prompt hashes, lane refs, output refs, and verdict attribution even when the concrete provider-qualified model id is the same.

### Runtime-Conformance Areas To Prove While Building

1. Actual OpenCode `subtask: true` lane lifecycle observability remains uncertain until exercised against the active OpenCode SDK/runtime and recorded as evidence.
2. Injected SDK reviewer lane launch is expected to be feasible, but the exact available runtime echo fields, parent/subtask identifiers, timeout semantics, and failure metadata depend on OpenCode API behavior.
3. Provider/model/account/auth binding can be enforced by FlowDesk policy, but how much of that binding can be independently echoed by OpenCode or external auth plugins remains an evidence question.
4. SDK-scoped cancellation/no-reply controls may be implemented only through first-class OpenCode SDK/plugin surfaces and explicit FlowDesk decisions; this is now complete for `permission.ask`, `session.abort`, and SDK prompt `noReply`. ### Non-Blocking Rule

If a Release 2/2.5 feature works in practice but a conformance proof is incomplete, do not block implementation solely for that reason. Instead, keep the feature opt-in, preserve default Release 1 non-dispatch behavior, store the best available durable evidence, mark the uncertain part in doctor/status/debug output, add negative tests for the safety boundary, and record the exact missing proof in this snapshot or a linked conformance note.
