# OpenCode 1.14.40 Conformance PoC Summary

Source report reference: `local-poc-report-2026-05-15-opencode-1.14.40`

Date: 2026-05-15

Scope: isolated local PoC with temporary workspace and OpenCode home directories. Raw local paths are intentionally omitted from this persisted summary.

## Verdict

This PoC is sufficient to support the current FlowDesk Release 1 target as a **General-Use MVP with chat-routed command-backed workflows**. The original PoC wording used “command-first developer preview” language because it was written during a narrower design step. Current product interpretation supersedes that wording: ordinary users start in chat, and accepted requests route into guarded command-backed workflows, with commands retained for setup, status, recovery, diagnostics, and fallback.

This PoC does not prove plugin-level automatic cross-provider fallback, managed provider/model reselection, or trusted real-provider runtime echo. Release 1 provider/API/model outage handling is limited to diagnostics, status, degraded behavior, guarded dry-run, and fake-runtime paths.

Supported by evidence:

1. Portable command-backed flows are PASS candidates for Release 1 controls and fallback.
2. Chat steering/routing is a PARTIAL/PASS candidate through `chat.message` mutation and throw behavior, sufficient for chat-routed command-backed workflows but not hard managed chat control.
3. Guard denial for model-driven built-in `bash` calls is PASS for the tested path and produced zero side effect.
4. `chat.params` and `chat.headers` reached a fake OpenAI-compatible provider, including the `x-flowdesk-conformance: 1` header.
5. Plugin hook and event surfaces are useful for situational awareness where the pinned version exposes session, message, tool, command, or related observations.

Still not proven or still gated:

1. Hard no-reply/cancel/stop authority for chat turns.
2. Trusted runtime echo against a real external provider.
3. General production-grade real dispatch beyond the scoped local PoC.
4. Sufficient correlated event telemetry for real dispatch, event-dependent recovery, stall detection, timeout handling, failure detection, and artifact quarantine.
5. Custom plugin tool schema compatibility, because PoC custom tool definitions caused a schema conversion crash before provider dispatch.
6. Command-level `agent`, `model`, and `subtask` routing/binding behavior.
7. Automatic provider/model fallback or reselection.
8. Provider Health Snapshot readiness for real dispatch.

## Evidence Highlights

1. Isolated OpenCode paths resolved under the PoC root.
2. Isolated config loaded only the local file plugin.
3. Command hooks fired for `dex-doctor`, `dex-plan`, `dex-run`, and related command probes. These were historical local probe names from the isolated PoC, not FlowDesk target command names.
4. `$ARGUMENTS` works; `$1` expands from positional content; `$name` remains literal.
5. `command.execute.before` mutation changes outgoing text and can steer command-backed flows.
6. `command.execute.before` throw prevents downstream chat dispatch.
7. `chat.message` mutation changes outgoing text and can steer normal chat requests toward a FlowDesk envelope.
8. `chat.message` throw aborts dispatch.
9. Unsupported `noReply`, `cancel`, and `stop` fields are not safe cancellation mechanisms.
10. Command template shell interpolation can execute before `tool.execute.before`; FlowDesk must not generate or permit user-controlled command templates containing shell interpolation.
11. Model-driven built-in `bash` tool calls triggered `tool.execute.before` and were denied with no marker file side effect.
12. The fake provider received `chat.params` and `chat.headers` output.
13. Canary scan found the fake canary only in plugin source, not in logs or provider traffic.
14. No evidence showed that plugin `event` observations can approve retry, fallback, dispatch, or provider/model reselection.

## Design Implications

1. Release 1 may use chat as the normal entry point and include chat steering/routing mode, but not hard-interception chat mode.
2. Release 1 may use command mutation and command throw as safe control mechanisms for command-backed flows.
3. Release 1 may use `chat.message` mutation/throw to wrap or steer normal-user requests into a FlowDesk envelope, as long as the product does not claim hard no-reply/cancel authority.
4. Guard denial via `tool.execute.before` is credible for tested model-driven built-in tool calls, but all privileged paths still require Guard and audit.
5. Real external provider runtime echo remains a later release gate.
6. Provider health is a separate diagnostic snapshot from usage availability. Release 1 may surface provider health and failure classes, but it must not use them to switch providers/models automatically.
7. FlowDesk command templates must be generated by FlowDesk only and must not include shell interpolation from user-controlled input.

## Original Wording Versus Current Release Interpretation

The PoC evidence originally supported “command-first developer preview plus chat steering/routing” as a cautious intermediate phrase. That phrase is historical. It should not be read as the current Release 1 product target.

Current Release 1 target:

1. General-Use MVP for ordinary OpenCode users.
2. Chat as the normal entry point for user requests.
3. Accepted chat requests route into guarded command-backed workflows.
4. Guarded dry-run and fake-runtime dispatch.
5. Guard-denied built-in tool side-effect tests.
6. Fake-provider header/params echo tests.
7. No hard no-reply/cancel claim.
8. No real external provider trusted echo claim.
9. No claim of real OpenCode dispatch until trusted binding, trusted runtime echo, sufficient event telemetry, fresh usage, fresh provider health, Guard approval, and durable pre-dispatch audit all pass.
10. No claim of managed provider/model fallback until fresh provider-native usage, fresh provider health, runtime compatibility, policy eligibility, trusted binding/echo, sufficient telemetry, durable pre-dispatch audit, a new attempt id, and explicit Guard approval all pass.
