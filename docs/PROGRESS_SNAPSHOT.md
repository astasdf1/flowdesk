# FlowDesk Progress Snapshot

## Purpose

This document tracks implementation progress against the full FlowDesk roadmap. Update it whenever work changes implementation status, release gates, user-facing readiness, or known blockers.

This snapshot is not a substitute for tests, conformance evidence, or release approval. It is a planning artifact that keeps the roadmap state visible between sessions.

## Update Rule

Every non-trivial FlowDesk work session must update this file before concluding if any of the following changed:

1. Code, tests, docs, packaging, installer behavior, conformance evidence, or release gates.
2. A phase’s estimated progress, blocker list, or readiness interpretation.
3. The distinction between prototype, local adapter, production registration, and user-ready behavior.
4. Any critical review finding that changes plan priority or scope.

If no progress fields changed, the final response should explicitly say the progress snapshot was checked and did not need an update.

## Release 2/2.5 Implementation Confidence and Uncertainty Policy

Release 2 and Release 2.5 work should proceed incrementally even when some OpenCode runtime conformance details are not yet fully proven. Lack of complete proof is not, by itself, a blocker when the feature can run safely behind explicit opt-in gates, fail-closed checks, durable evidence capture, and clear doctor/status reporting. Uncertain behavior must be documented as an evidence limitation and kept out of default Release 1 authority claims.

### Expected Implementable Areas

1. Durable production-path evidence persistence for usage authority, runtime echo, telemetry correlation, configured verification results, approval decisions, and pre-dispatch audit records is implementable using the existing `session-evidence.ts`, state-store, audit, and validator patterns.
2. A fail-closed production approval state machine is implementable as a core contract before dispatch is enabled; it should default to deny and only later permit conditional dispatch when required evidence is present.
3. Doctor-visible production enablement states are implementable: `disabled`, `blocked`, `configured`, `approved`, and `dispatch-capable` should be represented as diagnostic state, not as implicit dispatch authorization.
4. External auth/provider policy and doctor gating are implementable with conservative provider/model/account/auth-scope binding checks and explicit non-dispatchable diagnostics when evidence is missing or stale.
5. Configured verification integration is implementable by requiring production-path verification config to be read, validated, persisted, and treated as blocking only for the guarded dispatch path.
6. Same-model multi-perspective reviewer separation is implementable at the FlowDesk artifact layer by storing distinct perspective bindings, prompt hashes, lane refs, output refs, and verdict attribution even when the concrete provider-qualified model id is the same.

### Runtime-Conformance Areas To Prove While Building

1. Actual OpenCode `subtask: true` lane lifecycle observability remains uncertain until exercised against the active OpenCode SDK/runtime and recorded as evidence.
2. Injected SDK reviewer lane launch is expected to be feasible, but the exact available runtime echo fields, parent/subtask identifiers, timeout semantics, and failure metadata depend on OpenCode API behavior.
3. Provider/model/account/auth binding can be enforced by FlowDesk policy, but how much of that binding can be independently echoed by OpenCode or external auth plugins remains an evidence question.
4. SDK-scoped cancellation/no-reply controls may be implemented only through first-class OpenCode SDK/plugin surfaces and explicit FlowDesk decisions; this is now complete for `permission.ask`, `session.abort`, and SDK prompt `noReply`. Hook-level `chat.message` blocking/no-reply/cancel authority must remain explicitly unclaimed unless OpenCode exposes a first-class hook boundary. If that hook boundary is not available, FlowDesk may still run SDK-scoped control and steering features while documenting the hook limitation.
5. Runtime telemetry may require FlowDesk-created correlation artifacts if OpenCode does not expose all desired lane observability fields directly.

### Non-Blocking Rule

If a Release 2/2.5 feature works in practice but a conformance proof is incomplete, do not block implementation solely for that reason. Instead, keep the feature opt-in, preserve default Release 1 non-dispatch behavior, store the best available durable evidence, mark the uncertain part in doctor/status/debug output, add negative tests for the safety boundary, and record the exact missing proof in this snapshot or a linked conformance note.
